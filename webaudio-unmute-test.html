webaudio-unmute-test

Does iOS unmute Web Audio on the speaker when an HTMLAudioElement is also playing? If yes, t0ck can switch sounds instantly. If no, we're done tinkering and v47 is the best we get.
Test 1

Baseline: Web Audio alone

Plays 3 beeps using only Web Audio. On iOS speaker this should be silent (the bug we're working around in t0ck). If you hear anything, you're on headphones — switch to speaker and retest.
Test 2

HTMLAudio alone (control)

Plays a 3-second tone via an HTMLAudioElement. Should be audible on speaker — that's why t0ck v47 uses this path. Just confirming the control works.
Test 3

The hypothesis: HTMLAudio + Web Audio layered

Starts a silent-ish HTMLAudioElement track playing in the background, then fires a Web Audio beep on top. The question: does iOS "unmute" Web Audio because another media element has already established speaker output? If you hear the beep, the hybrid engine is viable. If only the background track comes through (no beep on top), the bug persists.
If Test 3 works, try tapping "Start + Beep" a few times in quick succession — the multiple beeps tell you whether Web Audio is actually driving the speaker, since only Web Audio could fire them that fast in that pattern.
Test 4

Variant: AudioContext created AFTER audio element plays

Same as Test 3, but we wait until the HTMLAudioElement is confirmed playing before we create the AudioContext. Some iOS Safari versions have different rules depending on the order.
